Module 1: Thinking Serverless

Introduction to Serverless Computing

What is Serverless?

  • No server management - Focus on code, not infrastructure
  • Automatic scaling - Scale from zero to thousands of requests
  • Pay-per-use - Only pay for actual compute time
  • Built-in availability - High availability and fault tolerance

Serverless vs Traditional

  • Reduced operational overhead
  • Faster time to market
  • Improved cost efficiency
  • Enhanced developer productivity

Serverless Computing Benefits

Cost Optimization

Pay only for requests and compute time used - no idle server costs

Automatic Scaling

Handle any number of requests without capacity planning

Faster Development

Focus on business logic instead of infrastructure management

Modern Serverless Application Best Practices

Architecture Principles

  • Single Responsibility - Each function has one clear purpose
  • Stateless Design - No persistent state between invocations
  • Event-Driven - React to events rather than polling
  • Microservices - Small, independent, loosely coupled services

Development Best Practices

  • Keep functions small and focused
  • Minimize cold start impacts
  • Use appropriate memory allocation
  • Implement proper error handling
  • Apply security by design

Design Patterns

Fan-Out Pattern

Single event triggers multiple independent processing functions

Aggregator Pattern

Collect and combine data from multiple sources

Circuit Breaker

Prevent cascading failures with automatic fallback mechanisms

Saga Pattern

Manage distributed transactions across multiple services

Module 2: API-Driven Development

Synchronous Event Sources Overview

Synchronous vs Asynchronous

Synchronous Processing

Request-Response Pattern - Client waits for immediate response from server

Asynchronous Processing

Fire-and-Forget Pattern - Client sends request and continues without waiting

When to Use Synchronous

  • User-facing APIs requiring immediate response
  • Real-time data retrieval
  • Interactive web applications
  • Mobile app backends

API-Driven Development Benefits

  • Clear interface contracts
  • Technology-agnostic integration
  • Independent team development
  • Easier testing and debugging
  • Better documentation and discovery

Standard Request/Response Flow

API Request Lifecycle

  1. Client sends HTTP request to API Gateway
  2. API Gateway validates and transforms request
  3. Lambda function processes business logic
  4. Response returned through API Gateway
  5. Client receives formatted response

Amazon API Gateway Integration with Serverless

API Gateway Core Features

  • Request/Response Transformation - Modify data format
  • Authentication & Authorization - Secure API access
  • Rate Limiting & Throttling - Control request volume
  • Caching - Improve performance and reduce costs
  • Monitoring & Logging - Track API usage and performance

Integration Types

  • Lambda Proxy: Direct Lambda integration
  • Lambda Custom: Transform request/response
  • HTTP Proxy: Forward to HTTP endpoints
  • AWS Service: Direct AWS service integration
  • Mock: Return static responses

API Gateway Request Processing

Request Flow

  1. Receive and validate HTTP request
  2. Apply authorization and authentication
  3. Check rate limits and quotas
  4. Transform request if needed
  5. Route to backend integration
  6. Process backend response
  7. Transform response if needed
  8. Return response to client

Security Importance

  • Data Protection - Safeguard sensitive information

Serverless Security Challenges

Security Layers

Infrastructure Security

AWS-managed infrastructure, VPC isolation, network security

Identity & Access Management

Application Security

Input validation, output encoding, secure coding practices

Data Security

Encryption at rest, in transit, and key management

Understanding Authentication vs Authorization

Authentication: "Who are you?"

Identity verification - Confirming user identity through credentials, tokens, or biometric data

Authentication Methods

  • Username/Password: Traditional credential-based
  • Multi-Factor Authentication: Additional security layers
  • OAuth/OpenID Connect: Token-based authentication
  • JWT Tokens: Self-contained authentication tokens
  • API Keys: Simple application authentication
  • Certificate-based: PKI infrastructure authentication

Authorization: "What can you do?"

Access control - Determining user permissions and resource access rights after authentication

Authorization Models

  • Role-Based Access Control (RBAC): Permissions based on roles
  • Attribute-Based Access Control (ABAC): Dynamic permissions
  • Resource-Based Policies: Direct resource permissions
  • Fine-Grained Permissions: Specific action-level control

Best Practice

Implement both authentication and authorization for comprehensive security

API Gateway Authentication Options

Custom authorization logic using Lambda functions

  • Token-based authorization

API Keys

Simple key-based access control for applications

Imperative vs Declarative Programming

Imperative Approach

"How to do it" - Step-by-step instructions specifying the exact process

# Imperative Example aws lambda create-function ... aws apigateway create-rest-api ... aws apigateway create-resource ...

Declarative Approach

"What you want" - Desired end state defined in templates

# Declarative Example Resources: MyFunction: Type: AWS::Lambda::Function MyAPI: Type: AWS::ApiGateway::RestApi

Module 5: Decoupling Components with EventBridge and SNS

Asynchronous Event Sources Development

Asynchronous vs Synchronous Patterns

Synchronous Processing

Client waits for immediate response, tight coupling, potential bottlenecks

Asynchronous Processing

Fire-and-forget pattern, loose coupling, improved resilience

When to Choose Asynchronous

  • Background processing tasks
  • Event-driven workflows
  • System integration scenarios
  • High-volume data processing
  • Microservices communication

Asynchronous Design Considerations

  • Loose Coupling - Services operate independently
  • Resilience - Failure isolation and recovery
  • Scalability - Independent scaling of components
  • Event Ordering - Managing message sequence when required
  • Error Handling - Dead letter queues and retry logic
  • Idempotency - Handle duplicate message processing

Trade-offs

Asynchronous patterns increase complexity but provide better scalability, resilience, and performance

  • Targets - Destinations for matched events

Advanced Features

  • Archive and replay capabilities

Module 6: Event-Driven Development Using Queues and Streams

Polling Event Sources for Lambda Triggers

Polling vs Push Model

Push Model (Synchronous)

Event source invokes Lambda - API Gateway, S3, EventBridge directly trigger functions

Polling Model (Asynchronous)

Lambda polls event source - Lambda service reads from queues and streams

Polling Event Sources

  • Amazon SQS (Standard and FIFO queues)
  • Amazon DynamoDB Streams
  • Amazon Kinesis Data Streams
  • Amazon MSK (Managed Kafka)
  • Amazon MQ
  • Self-managed Apache Kafka

Event Source Mapping

SQS Queue
→
Lambda Poller
→
Function

Processing Characteristics

  • Batch Processing - Process multiple records together
  • Parallel Execution - Multiple concurrent function invocations
  • Error Handling - Automatic retries and DLQ integration
  • Scaling Control - Configure concurrency and batch size

Configuration Options

  • Batch size (1-10 for SQS, 1-100 for streams)
  • Maximum batching window
  • Parallel execution factor
  • Error handling and retry settings

Understanding Queues vs Streams

Queue Characteristics (SQS)

Message Durability

At-least-once delivery - Messages persist until successfully processed

SQS Features

  • Variable processing order (Standard queues)
  • Exactly-once processing (FIFO queues)
  • Message visibility timeout
  • Dead letter queue support
  • Long polling for efficiency
  • Message deduplication

Queue Use Cases

  • Decoupling application components
  • Background job processing
  • Load leveling and buffering
  • Workflow orchestration

Stream Characteristics

Ordered Processing

Sequential record processing - Maintains order within shards or partitions

Stream Features

  • Time-ordered sequence of records
  • Multiple consumers can read same stream
  • Configurable retention period
  • Shard-based parallel processing
  • Real-time data processing
  • Checkpoint-based progress tracking

Stream Use Cases

  • Real-time analytics
  • Event sourcing patterns
  • Change data capture
  • IoT data processing
  • Log aggregation
